# EXAMPLES — Agent Team 协作

## 反例 1: 主线程独自吞下一个跨 8 文件的任务

**Anti-pattern**:

用户: "给整个项目把 SQLite 迁到 PostgreSQL"

主线程行为(错误): 直接在主对话里逐文件 Read + Edit,读了 12 个文件、改了 8 个,上下文一路冲到 200k token,后续连用户原始需求都记不清了。

**问题**:
- 主上下文被实现细节(SQL 字符串、SQLAlchemy session、migration script)塞满
- 用户看不到分阶段进度,要在巨大输出里找重点
- 没人验收,迁完就说"完成"

**Pattern (正确)**:

主线程做的事:
1. 规划: "拆 3 个子任务: (A) DDL + schema 迁移脚本 (B) ORM 切换 (C) 测试套件适配"
2. 派工程师 agent: "做 A,只动 `migrations/` 目录,完成后给我 schema diff + 测试结论"
3. 派监理 agent: "对 A 验收 ✅/❌/⚠️,重点查 nullable / 索引 / 外键的迁移是否丢失"
4. 监理过了 → 进入 B,同样套路
5. 主线程冒烟测试: 起服务、点登录、看日志,最后给用户一份合并报告

```
主线程上下文: ~30k token,只有规划 + 监理结论 + 冒烟脚本输出
子 agent 上下文: 各 100k token 但独立窗口,用完即弃
```

## 反例 2: 跳过监理,工程师自报"完成"

**Anti-pattern**:

主线程: 派工程师 agent 写一个新 REST API。
工程师 agent 返回: "代码已写,完成。"
主线程: 直接告诉用户"完成"。

**问题**: 没人扮演监理。工程师 agent 会倾向证明自己干的对,缺乏对抗性测试。

**Pattern (正确)**:

工程师 agent 返回 → 主线程 **必须** 再开一个监理 agent:

> 监理任务: 对 `POST /api/users` 接口做对抗性测试
> - 正确路径: 有效输入返回 201
> - 错误路径: 重复 email / 缺字段 / 超长字段 / SQL 注入 / 大 payload
> - 边界: email 长度 = 254 / 255 / 256
> - 并发: 同时 POST 同一 email
> 输出 ✅/❌/⚠️,FAIL 列出哪条复现命令

如果监理 FAIL → 回工程师修 → 再过监理。绿了才算完。

## 反例 3: 把"沟通"也派给 agent

**Anti-pattern**:

主线程把 "回复用户为什么这个方案要这么选" 派给一个 sub-agent 起草,然后转贴。

**问题**: 沟通是主线程的不可外包职责。用户问 "为什么",要的是和当前主线程上下文一致的答案,sub-agent 没有这个上下文。

**Pattern (正确)**:

| 工作类型 | 谁干 |
|---|---|
| 写代码 / 改代码 / 跨多文件编辑 | 工程师 agent |
| 跑测试 / 找 bug / 边界值验证 | 监理 agent |
| 调研某个库 / 搜索 codebase / 列依赖 | research agent |
| 规划阶段拆 / 跨阶段汇总 / 跟用户对话 | 主线程(不外包) |
| 验收监理结论 / 决定下一步 | 主线程(不外包) |
| 冒烟测试 / 起服务点一遍 / 看用户原始诉求是否满足 | 主线程(不外包) |
